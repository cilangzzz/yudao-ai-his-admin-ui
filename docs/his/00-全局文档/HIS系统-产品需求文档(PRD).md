## 1. 需求概述

### 1.1 需求背景

随着医疗信息化建设的深入推进，三级及以上医院亟需一套符合国家标准、具备AI辅助能力的综合医院信息系统。当前市场上的HIS系统普遍存在以下问题：

- 系统架构老旧，难以扩展和集成
- 不符合HL7 FHIR等国际互操作标准
- 缺乏闭环药物管理和临床决策支持能力
- AI辅助功能薄弱，无法满足智慧医疗需求

本项目基于YUDAO（若依）框架，构建新一代智慧医疗HIS系统，目标达到HIMSS EMRAM Stage 5+水平。

### 1.2 需求目标

| 目标类型 | 目标描述 | 衡量指标 |
|----------|----------|----------|
| 业务目标 | 全面覆盖三级医院核心业务流程 | 13个子系统100%覆盖 |
| 业务目标 | 符合国家医院信息化建设标准 | 通过互联互通标准化成熟度测评 |
| 用户目标 | 提升医护人员工作效率 | 平均就诊时间 ≤ 30分钟 |
| 用户目标 | 保障患者安全和用药安全 | 闭环给药核对率 = 100% |
| 技术目标 | 实现系统间互联互通 | 基于HL7 FHIR R4的10+外部接口 |
| 技术目标 | 满足等保三级安全要求 | 通过等保三级认证 |

### 1.3 预期收益

- **直接收益**: 门诊效率提升30%，医嘱执行及时率≥95%，危急值通报≤15分钟
- **间接收益**: 医疗差错率降低90%（闭环给药），病历质控合格率提升至98%
- **长期收益**: 数据资产积累，支撑科研和运营决策

### 3.2 P0核心功能详情（第一期MVP）

#### 3.2.1 M09 系统管理子系统

##### F-SYS-001: 用户与权限管理 (RBAC)

**功能描述**: 基于RBAC模型实现用户账号、角色、权限的细粒度管理。

**业务规则**:
- BR-SYS-001: 用户账号必须唯一，支持用户名/工号登录
- BR-SYS-002: 角色权限支持菜单权限、按钮权限、数据权限三级控制
- BR-SYS-003: 密码必须满足复杂度要求（8位以上，含大小写字母+数字+特殊字符）
- BR-SYS-004: 连续登录失败5次锁定账户30分钟
- BR-SYS-005: 敏感操作必须二次确认

**RICE评分**: 333（影响全院，工作量低）

##### F-SYS-002: 数据字典管理

**功能描述**: 统一管理系统基础数据字典，包括诊断编码(ICD-10)、药品目录、收费项目等。

**业务规则**:
- BR-SYS-006: 数据字典变更必须记录版本历史
- BR-SYS-007: 被引用的字典项不可删除，只能停用

##### F-SYS-003: 审计日志

**功能描述**: 记录所有关键操作的审计日志，支持操作回溯。

**业务规则**:
- BR-SYS-008: 审计日志不可删除、不可修改
- BR-SYS-009: 敏感数据操作必须记录操作前后值
- BR-SYS-010: 日志保留期限≥3年（等保要求）

#### 3.2.2 M01 门诊管理子系统

##### F-OP-001: 挂号管理

**功能描述**: 支持现场挂号、预约挂号（微信/APP/网站）、自助机挂号、急诊挂号。

**业务规则**:
- BR-OP-001: 预约挂号支持7天内号源，每人每科室每日限约1次
- BR-OP-002: 退号需在就诊前办理，已就诊不可退号
- BR-OP-003: 急诊挂号自动按病情分级分诊（Ⅰ-Ⅳ级）
- BR-OP-004: 医保挂号需实时验证医保身份

**用户旅程**:
```
患者到达 → 选择挂号方式 → 选择科室/医生 → 缴纳挂号费 → 分配排队号 → 候诊
```

**RICE评分**: 178

##### F-OP-002: 门诊医生工作站

**功能描述**: 为门诊医生提供接诊、开方、申请检查检验的工作平台。

**业务规则**:
- BR-OP-005: 处方开立时必须进行CDS校验（药物相互作用、过敏检查、剂量合理性）
- BR-OP-006: 诊断必须使用ICD-10编码
- BR-OP-007: 处方金额超过500元需二次确认
- BR-OP-008: 门诊病历必须在就诊完成后24小时内完成

**CDS（临床决策支持）要求**:
- 药物-药物相互作用自动检查
- 基于患者过敏史的药物过敏警告
- 基于年龄/体重/肾功能的剂量合理性提示
- 诊断与检验检查结果一致性校验

**RICE评分**: 178

##### F-OP-003: 门诊收费管理

**功能描述**: 自动汇总处方/检查/检验费用，支持多种支付方式和医保实时结算。

**业务规则**:
- BR-OP-009: 收费前必须核对患者身份
- BR-OP-010: 医保患者必须实时对接医保系统结算
- BR-OP-011: 退费必须有审批流程，部分退费需逐项确认
- BR-OP-012: 收费员每日必须日结对账

##### F-OP-004: 门诊药房管理

**功能描述**: 处方接收、审核、调配、发药的全流程管理。

**业务规则**:
- BR-OP-013: 发药前必须核对患者身份（姓名+就诊号）
- BR-OP-014: 麻醉药品、精神药品必须专项管理（五专管理）
- BR-OP-015: 药品库存低于安全库存时自动预警

#### 3.2.3 M02 住院管理子系统

##### F-IP-001: 入院管理

**功能描述**: 入院登记、床位分配、预交金管理、医保登记。

**业务规则**:
- BR-IP-001: 入院必须关联门诊诊断或急诊诊断
- BR-IP-002: 预交金不足时自动提醒，欠费超过阈值限制记账
- BR-IP-003: 床位分配需考虑性别、病种、感染等因素
- BR-IP-004: 入院护理评估（跌倒、压疮风险）必须在入院4小时内完成

##### F-IP-002: 住院医生工作站

**功能描述**: 医嘱管理（长期/临时）、病历书写、诊断管理、手术申请。

**业务规则**:
- BR-IP-005: 医嘱开立必须进行CDS校验（同门诊）
- BR-IP-006: 长期医嘱停止需提前一天，临时医嘱立即执行
- BR-IP-007: 医嘱状态流转：开立→审核→执行→完成
- BR-IP-008: 入院记录必须24小时内完成，首次病程记录8小时内完成
- BR-IP-009: 知情同意书必须电子签名后方可执行

##### F-IP-003: 护理工作站

**功能描述**: 医嘱执行、护理记录、体温单、护理评估、eMAR给药记录。

**业务规则**:
- BR-IP-010: 给药前必须扫描患者腕带和药品条码（闭环给药，HIMSS EMRAM Stage 5）
- BR-IP-011: 腕带或条码不匹配时必须停止给药并核实
- BR-IP-012: 生命体征异常时必须立即通知医生
- BR-IP-013: 交接班必须完成交接班记录

##### F-IP-004: 出院管理

**功能描述**: 出院申请、出院结算、出院带药、病案归档。

**业务规则**:
- BR-IP-014: 出院前必须停止所有长期医嘱
- BR-IP-015: 出院结算必须完成费用审核，无未结费用
- BR-IP-016: 医保患者出院必须完成医保结算
- BR-IP-017: 病案归档后不可修改

#### 3.2.4 M06 药品管理子系统

##### F-PHARM-001: 药库管理

**功能描述**: 药品入库、出库、盘点、效期管理、批号追踪。

**业务规则**:
- BR-PHARM-001: 药品入库必须验收（品名、规格、批号、效期、数量）
- BR-PHARM-002: 近效期药品（≤3个月）自动预警
- BR-PHARM-003: 药品出库遵循"先进先出"原则
- BR-PHARM-004: 麻醉药品必须"五专"管理（专人、专柜、专锁、专账、专方）

##### F-PHARM-002: 处方审核与合理用药

**功能描述**: 处方合理性自动审核，药物相互作用、配伍禁忌检查。

**业务规则**:
- BR-PHARM-005: 所有处方必须经过合理性审核方可调配
- BR-PHARM-006: 药物-药物相互作用检查必须覆盖所有已知高风险组合
- BR-PHARM-007: 抗菌药物使用需符合分级管理规定

**RICE评分**: 225

### 3.3 P1重要功能详情（第二期）

#### 3.3.2 M04 检验管理子系统 (LIS)

##### F-LIS-001: 检验全流程管理

**功能描述**: 检验申请→标本采集→检验→结果审核→报告发布的全流程管理。

**业务规则**:
- BR-LIS-001: 标本必须全程条码追踪（采集→运送→接收→检验→报告）

- BR-LIS-003: 常规检验报告时间≤2小时
- BR-LIS-004: 检验结果异常标志必须自动标识（偏高/偏低/危急值）

**RICE评分**: 170

#### 3.3.3 M05 影像管理子系统 (RIS/PACS)

##### F-RIS-001: 影像全流程管理

**功能描述**: 基于DICOM标准的影像检查全流程管理。

**业务规则**:
- BR-RIS-001: 影像必须通过DICOM标准接口采集和存储
- BR-RIS-002: 影像存储采用三级架构（在线/近线/离线）
- BR-RIS-003: 影像查看器必须支持测量、标注、窗宽窗位调整
- BR-RIS-004: 常规检查报告时间≤4小时

#### 3.3.4 M08 财务管理子系统

**业务规则**:

- BR-FIN-002: 财务操作必须审计追踪
- BR-FIN-003: 收费项目价格调整必须记录历史

|------|------|----------|
| 常规操作响应时间 | ≤ 2秒 | JMeter压力测试 |
| 复杂查询响应时间 | ≤ 5秒 | JMeter压力测试 |
| 并发用户数 | ≥ 1000 | 负载测试 |
| 日门诊量 | ≥ 5000人次 | 容量测试 |
| 影像加载时间 | ≤ 3秒（常规）/ ≤ 10秒（大型） | 性能测试 |
| 报表生成时间 | 日报≤30秒，月报≤3分钟 | 性能测试 |

### 4.2 安全需求

| 要求 | 具体标准 |
|------|----------|
| 身份认证 | 用户名密码/CA证书/LDAP多种方式 |
| 权限控制 | RBAC细粒度权限+数据权限 |
| 数据加密 | 敏感数据加密存储，传输TLS 1.2+ |
| 审计追踪 | 所有关键操作留痕 |
| 数据备份 | RPO ≤ 1小时，RTO ≤ 4小时 |
| 患者隐私 | 符合《个人信息保护法》，数据脱敏 |
| 等保要求 | 信息安全等级保护三级 |

### 4.3 可用性需求

| 指标 | 要求 |
|------|------|
| 系统可用率 | ≥ 99.9%（年停机≤8.76小时） |
| 故障自动切换 | ≤ 30秒 |
| 数据丢失概率 | ≤ 10⁻⁹ |

### 4.4 兼容性需求

- **浏览器**: Chrome 80+, Firefox 75+, Edge 80+, Safari 13+
- **分辨率**: 1920×1080及以上，响应式布局
- **服务端**: Linux (CentOS 7+/Ubuntu 18.04+), Windows Server 2016+

### 4.5 可扩展性需求

- 微服务架构，各业务模块独立部署

## 8. 交互设计

### 8.1 关键交互流程

#### 门诊就诊主流程
```
患者到达 → 挂号/签到 → 候诊叫号 → 医生接诊
  → 开立诊断(ICD-10) → 开具处方 → CDS自动校验
  → 患者缴费 → 药房取药 → 就诊结束
```

#### 闭环给药流程（HIMSS EMRAM Stage 5）
```
医嘱开立 → 药师审核 → 药房配药(条码贴签)
  → 配送至病区 → 护士接收核对
  → 扫描患者腕带 → 扫描药品条码
  → 身份+药品双重匹配 → 确认给药 → 记录eMAR
```

#### 医嘱处理流程
```
医生开立医嘱 → CDS自动校验 → 医嘱提交
  → 护士站接收 → 护士审核 → 医嘱执行 → 执行记录
```

### 8.2 状态转换

#### 医嘱状态流转
| 当前状态 | 触发条件 | 目标状态 | 说明 |
|----------|----------|----------|------|
| 开立 | 医生提交 | 审核 | 等待护士审核 |
| 审核 | 护士审核通过 | 执行中 | 开始执行 |
| 审核 | 护士退回 | 开立 | 退回修改 |
| 执行中 | 执行完成 | 已完成 | 记录执行结果 |
| 开立/审核 | 医生作废 | 已作废 | 作废医嘱 |
| 审核/执行中 | 医生停止 | 已停止 | 停止长期医嘱 |

#### 处方状态流转
| 当前状态 | 触发条件 | 目标状态 |
|----------|----------|----------|
| 开立 | 医生提交处方 | 审核中 |
| 审核中 | 药师审核通过 | 审核通过 |
| 审核中 | 药师退回 | 审核退回 |
| 审核通过 | 药师调配 | 已调配 |
| 已调配 | 发药确认 | 已发药 |
| 已发药 | 退药申请 | 已退药 |

## 9. 验收标准

### 9.1 功能验收

采用Gherkin格式定义核心功能验收标准：

```gherkin
Feature: 门诊挂号

  Scenario: 现场挂号成功
    Given 患者持有有效身份证件
    And 系统中已有该患者档案
    When 收费员选择科室、医生、挂号类型
    And 收费员确认收取挂号费
    Then 系统生成挂号单
    And 分配排队序号
    And 患者进入候诊队列

  Scenario: 预约挂号成功
    Given 患者通过微信公众号登录
    And 选择7天内的号源
    When 患者确认预约并在线支付
    Then 系统生成预约挂号单
    And 发送预约成功短信

  Scenario: 医保挂号身份验证
    Given 患者持医保卡挂号
    When 收费员刷卡读取医保信息
    Then 系统实时验证医保身份
    And 按医保标准计算挂号费
```

```gherkin
Feature: 闭环给药（HIMSS EMRAM Stage 5）

  Scenario: 闭环给药成功
    Given 护士已接收待执行给药医嘱
    And 药品已调配并贴有条码
    When 护士扫描患者腕带
    And 腕带信息与医嘱匹配
    And 护士扫描药品条码
    And 药品信息与医嘱匹配
    Then 确认给药执行
    And 记录eMAR
    And 费用自动记账

  Scenario: 腕带不匹配停止给药
    Given 护士准备给药
    When 护士扫描患者腕带
    And 腕带信息与医嘱不匹配
    Then 系统停止给药流程
    And 显示警告信息"患者身份不匹配"
    And 要求护士核实患者身份

  Scenario: 药品不匹配停止给药
    Given 患者腕带验证通过
    When 护士扫描药品条码
    And 药品信息与医嘱不匹配
    Then 系统停止给药流程
    And 显示警告信息"药品不匹配"
    And 要求联系药师确认
```

```gherkin
Feature: 危急值管理

  Scenario: 危急值自动通报
    Given 检验结果出现危急值
    When 检验技师确认危急值结果
    Then 系统自动通知临床科室
    And 记录通知时间
    And 开始15分钟计时

  Scenario: 危急值超时未确认
    Given 危急值已通知临床科室
    When 超过15分钟未收到确认
    Then 系统升级通知科室主任
    And 记录超时事件
```
